차별점 1

전체 경고가 아니라 이번 변경의 위험을 봅니다

리포지토리 전체 노이즈보다, 지금 PR 때문에 새로 생긴 위험을 먼저 드러냅니다.

차별점 2

Next.js가 특히 잘 틀리는 지점을 노립니다

Server Actions drift, ISR/revalidate 충돌, standalone output 누락처럼 배포 직전 중요한 실수에 집중합니다.

차별점 3

숨긴 경고도 운영합니다

suppress를 그냥 묻어두지 않고, active, expired, invalid 상태와 예외 사유를 함께 다룹니다.

차별점 4

PR에서 바로 읽히는 결과를 남깁니다

팀이 새로운 UI를 배우지 않아도, 기존 머지 흐름 안에서 바로 사용할 수 있습니다.

경고를 숨기는 게 아니라, 예외를 운영합니다

같은 경고라도 왜 숨겨졌는지, 언제까지 허용되는지, 지금도 유효한지까지 함께 봐야 팀이 안심하고 머지할 수 있습니다. Next.js처럼 배포 실수가 치명적인 영역일수록 더 그렇습니다.

active, expired, invalid 상태를 분리하고, 어떤 규칙이 적용됐는지 추적 가능한 형태로 다룹니다.

name: Deploy Rehearsal
on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  contents: read
  pull-requests: write

jobs:
  rehearse:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # baseline 비교에 history 필요
      - uses: amu-create/deploy-rehearsal-mcp/action@v1

자주 묻는 질문

ESLint나 테스트와 뭐가 다른가요? ▾

이 도구는 코드 스타일이나 일반 오류보다, 실제 배포를 망가뜨릴 가능성이 큰 변경을 PR 기준으로 다시 봅니다. 전체 경고를 늘리는 게 아니라, 이번 변경의 위험을 줄이는 데 집중합니다.

왜 Next.js에 특히 강한가요? ▾

초기엔 Next.js/Vercel에서 자주 터지는 운영성 문제를 우선 겨냥했습니다. Server Actions, revalidate, standalone output처럼 배포 관점에서 중요한 실수를 바로 데모 가능한 형태로 잡아냅니다.

suppress를 왜 따로 강조하나요? ▾

실무에서는 경고를 아예 없애기보다 예외를 관리해야 하는 경우가 많습니다. 그래서 active, expired, invalid 상태와 적용 이유를 함께 보여주는 방향으로 설계했습니다.
